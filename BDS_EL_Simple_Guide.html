1. The Problem — In Plain English

Real-world analogy

Imagine a hospital wants to train an AI to detect diseases. To train it, the AI needs patient records. But handing over raw patient records violates privacy. Yet adding too much "scrambling" to the data makes the AI useless. That exact dilemma — in network security — is what we solve.

In our project:

  • The "patient records" are network traffic logs — 125,974 records of internet connections, each labelled as either "Normal" or an "Attack" (hacking attempt)
  • The "AI" is an Intrusion Detection System (IDS) — it learns to automatically flag suspicious traffic
  • The "privacy violation" is that network logs contain sensitive company data, user behaviour, and infrastructure details
The core problem

How do you train an AI on sensitive data — and still guarantee that no individual's information can ever be extracted from that AI — without destroying the AI's accuracy?

Three specific problems existed before our work:

  1. The original code was wrong. It added the same amount of random noise to every feature, regardless of what the data actually looked like. For some features this meant 250× too much noise (ruining accuracy). For others, not enough (still not private).
  2. Nobody documented what happens with many features. When you split the privacy budget across 37 features, each gets so little that the AI can't learn anything — it just guesses the most common answer. This is called "budget collapse" and had never been properly studied before.
  3. Privacy math was wasteful. Tracking how much privacy you've "spent" across many analyses was being done with loose approximations — wasting up to 3× the budget unnecessarily.

2. What We Set Out to Do (Objectives)

  1. Fix the broken foundation. Replace the wrong fixed-noise approach with one that measures actual data ranges and adds the correct amount of noise.
  2. Show the budget collapse problem exists and explain why. Run experiments proving that the standard approach fails on real data with 37 features, and formally explain the cause.
  3. Build a smarter approach (DP-SGD) that bypasses the problem entirely. Add noise to the AI's learning process instead of the data — this avoids the 37-feature problem completely.
  4. Introduce three new privacy techniques that reduce noise, improve efficiency, and track privacy cost more accurately — each with a formal mathematical proof.
  5. Validate everything rigorously — 30 repeated experiments, confidence intervals, and an actual attack on our own system to confirm it can't be broken.

3. How Privacy Works Here — Differential Privacy (Simple)

Think of it this way

Before publishing an election poll result, you add a tiny random number (say ±2%) to the final percentage. The real trend is still visible, but nobody can pin down exactly which voter answered what. That is essentially Differential Privacy — add calibrated random noise so the result is useful but no individual can be identified.

The key number is ε (epsilon) — the "privacy dial":

ε valuePrivacy levelNoise addedAccuracy
0.1Extremely strongEnormousAI fails completely
0.5Very strongHighSome loss
1.0Strong ✓Moderate94.70% (our result)
2.0GoodLowNear-perfect
10+WeakVery lowSame as no privacy

The goal is to operate at ε = 1.0 — strong, formally proven privacy — while keeping accuracy as close to the no-privacy baseline as possible.

5. Methodology — How We Did It (Step by Step)

Step 1 — Fix the Noise Calculation

Before adding any noise, you must know how sensitive the data is (how much one person's record can change a result). The original code assumed sensitivity = 1.0 for everything — completely wrong. We compute it from the actual data: Global Sensitivity = (max value − min value) / number of records. This alone fixed a 250× error.

Step 2 — Add Noise to Features (and Watch It Fail)

The classic approach: add Laplace random noise to each feature scaled to its sensitivity and privacy budget. This is called Input Perturbation. We ran this on all 37 features with ε=1.0. Each feature got ε/37 ≈ 0.027. The noise was so overwhelming the AI could only predict "Normal" for every input — accuracy collapsed to 53.3%. This is the budget collapse problem, now documented for the first time.

Step 3 — Apply DP-SGD (The Real Solution)

Instead of noising the data before training, we add noise during the AI's learning process — specifically to the "gradient" (the small update the AI makes each time it learns from a record). Because we clip each gradient to a fixed maximum length before adding noise, the sensitivity is a fixed constant regardless of how many features there are. Result: 94.70% accuracy at ε=1.0 — the budget collapse disappears entirely.

Step 4 — Apply Our Three Novel Techniques

On top of DP-SGD, we layered three new ideas (detailed in Section 7) that each reduce noise or improve privacy accounting.

Step 5 — Validate With an Attack

We built the most powerful possible attacker (a Membership Inference Attacker) to try to figure out if a specific person's record was in the training data. It failed every time — 0 out of 24 attempts violated our privacy guarantee. This proves the system works in practice, not just in theory.

7. Our Novel Contributions — What Is Actually New

These three things did not exist in the original code and represent our research contribution:

NOVEL CONTRIBUTION 1

Clipped Sensitivity — Reduce Required Noise by up to 592 Million Times

The problem: Some features have extreme outliers. Imagine one hacker who tried to switch to admin mode ("su" command) 10,000 times in one session. Everyone else tried it 0–2 times. That single person forces you to add enormous noise to protect them, hurting accuracy for everyone.

Our solution: Before measuring how sensitive the data is, clip it at the 99th percentile. If 99% of people tried "su" at most 3 times, we treat 3 as the maximum — not 10,000. This is mathematically valid because we formally prove the small bias this introduces is bounded and tiny.

Result: For the feature su_attempted, the required noise drops by 592,825,447 times (592 million×). The AI gets much more useful data while still being formally private.

NOVEL CONTRIBUTION 2

Importance-Weighted Budget Allocation — Smart Privacy Spending

The problem: If you split ε=1.0 equally across 37 features, each gets 0.027 — way too little. The AI collapses.

Our solution: Measure how useful each feature actually is for detecting attacks (using Mutual Information — how much knowing this feature tells you about whether it's an attack). Spend more privacy budget on useful features, almost nothing on useless ones.

Example: src_bytes (data sent) strongly predicts DoS attacks → gets a large budget share. num_outbound_cmds (almost always zero) → gets almost none.

Proven: By a theorem called Basic Composition, the total privacy cost still equals exactly ε=1.0. We just spent it smarter.

NOVEL CONTRIBUTION 3

PRV Accountant — Better Privacy Bookkeeping Using FFT Math

The problem: Every query you run on private data "costs" some privacy budget. After 100 queries, how much did you spend total? The standard method (RDP) overestimates by up to 2×, making you think you've spent twice as much as you actually have.

Our solution: Instead of a rough estimate, we track the exact distribution of privacy costs and combine 100 of them precisely using a mathematical technique called FFT (Fast Fourier Transform — the same math used in audio processing and signal analysis).

Result: After 100 queries at ε₀=0.5 each:
• Naive count says: ε_total = 50.0 (way off)
• Standard RDP method says: ε = 31.5
• Our PRV says: ε = 15.34 — 51.4% more accurate than RDP
This means you can run twice as many analyses for the same privacy guarantee.

11. Why Is This Research-Worthy?

Four reasons this work would be accepted at a top security or Big Data journal:

  1. We found and documented a new problem. Budget collapse in high-dimensional IDS is not well-studied. We prove it exists, explain why, and show that it affects all 5 attack classes (not just binary detection).
  2. Our solutions come with mathematical proofs. All three novel contributions (clipped sensitivity bias bound, composition theorem for weighted allocation, PRV tightening) are formally justified — not just "things that seem to work."
  3. We ran real statistical experiments. 30 runs per condition, 95% confidence intervals, Wilcoxon signed-rank significance tests. Most DP papers report single-run results — this is the rigor that top journals require.
  4. We validate against an actual attack. Zero bound violations in 24 MIA experiments confirms the system works in the real world, not just mathematically.
