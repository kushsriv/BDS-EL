# BDS-EL: Differential Privacy for Network Intrusion Detection

A reproducible **differential privacy (DP) research framework** applied to network intrusion
detection systems (IDS). Validated on **NSL-KDD** (125,974 records, 37 features) and
**UNSW-NB15** (82,332 records, 39 features) with 13 experiment modules, 3 novel algorithmic
contributions, an **Apache Spark distributed pipeline**, and a **live React dashboard**.

## Novel Contributions

| Contribution | File | Key Result |
|---|---|---|
| **Clipped Sensitivity** — clips features at the p-th percentile before computing GS; formally ε-DP with bounded bias | `sensitivity.py` | Up to **592,825,447× noise reduction** on `su_attempted` at p=99% |
| **Importance-Weighted Budget Allocation** — distributes ε proportional to MI / variance / SNR | `weighted_budget.py` | MI-weighted achieves **80.95%** vs 54.29% uniform at ε=5 |
| **PRV Accountant** — FFT convolution of privacy loss distributions (Gopi et al. 2021) | `prv_accountant.py` | **51% tighter** than RDP at k=100 queries (15.34 vs 31.54) |

## Key Results (30 runs, 95% CI)

### Binary IDS — NSL-KDD (37 features)

| ε | Uniform Input Pert. | MI-Weighted | **DP-SGD** |
|---|---|---|---|
| ∞ (no DP) | 94.77% | 94.77% | 94.77% |
| 0.1 | 54.28% | 53.29% | **92.80%** |
| 0.5 | 54.29% | 53.29% | **94.45%** |
| 1.0 | 54.29% | 53.29% | **94.70%** |
| 5.0 | 54.29% | 80.95% | — |

**DP-SGD at ε=1.0 loses only 0.07 pp** vs. the no-DP baseline.
Input perturbation collapses to majority-class prediction (53.3%) at all ε ≤ 5 because splitting
the budget across 37 features leaves only ε/37 ≈ 0.027 per feature — too noisy for any classifier.

### Cross-Dataset Validation (ε = 1.0)

| Dataset | No-DP | Input Perturbation | DP-SGD |
|---|---|---|---|
| NSL-KDD (37 feat.) | 94.77% | 54.29% | **94.70%** |
| UNSW-NB15 (39 feat.) | 92.27% | 55.00% | **90.22%** |

Budget collapse is **dataset-independent** — same problem appears on both benchmarks.

### Spark Pipeline — Budget Allocation at ε_total = 1.0

| Method | Top-feature ε | Min ε | Ratio |
|---|---|---|---|
| MI-Weighted | 0.0893 | 0.000270 | **331×** |
| SNR-Heuristic | 0.0697 | 0.000270 | **258×** |
| Variance-Weighted | 0.6742 | 0.000268 | **2515×** |
| Uniform | 0.0270 | 0.027027 | 1× |

### PRV vs. RDP Composition (Laplace, ε₀ = 0.5)

| k queries | RDP | **PRV** | Tightening |
|---|---|---|---|
| 10 | 6.28 | 3.89 | 38% |
| 50 | 21.46 | 11.01 | 49% |
| 100 | 31.54 | **15.34** | **51%** |

## Quick Start

### 1. Install Python dependencies

```bash
pip install -r requirements.txt
```

### 2. Run the pipeline

```bash
# With Spark distributed mode (requires Java 17 — see Spark Setup below)
python main.py --dataset KDDTrain+.csv --ml_runs 5

# Without Spark — pandas fallback, always works, same results
python main.py --dataset KDDTrain+.csv --ml_runs 5 --no_spark

# Quick test — skips slow modules (~2 min)
python main.py --dataset KDDTrain+.csv --ml_runs 2 --skip_multiclass --skip_budget_compare --skip_dpsgd --skip_prv --skip_stats
```

At the end of every run `export_results.py` runs automatically and writes
`frontend/public/results_data.json` — the dashboard picks it up on next refresh.

### 3. Launch the live dashboard

```bash
cd frontend
npm install
npm run dev
# Open http://localhost:5173
```

The hero badge shows **LIVE DATA · \<timestamp\>** once real results are loaded,
or **DEMO DATA** if no run has happened yet.

## Spark Setup (Windows — for Big Data distributed mode)

Spark needs Java 17. Java 25 has a removed API that breaks PySpark on Windows.

### Step 1 — Install Java 17

```powershell
winget install Microsoft.OpenJDK.17
```

### Step 2 — Find the exact install path

```powershell
Get-ChildItem "C:\Program Files" -Recurse -Filter "java.exe" -ErrorAction SilentlyContinue | Select-Object FullName
```

### Step 3 — Set JAVA_HOME (in each VS Code terminal session)

```powershell
$env:JAVA_HOME = "C:\Program Files\Microsoft\jdk-17.0.19.10-hotspot"   # use your actual path
$env:PATH = "$env:JAVA_HOME\bin;$env:PATH"
```

### Step 4 — Run

```powershell
cd BDS-EL
python main.py --dataset KDDTrain+.csv --ml_runs 5
```

### Make permanent (so you never set it again)

```powershell
[System.Environment]::SetEnvironmentVariable("JAVA_HOME", "C:\Program Files\Microsoft\jdk-17.0.19.10-hotspot", "User")
```

Restart VS Code after this.

## How Spark Works in This Project

```
main.py
  └─► _run_spark_dp_pipeline_module()
         └─► spark_pipeline.py
                ├─► SparkSession.builder.getOrCreate()    ← Java/JVM starts here
                │
                ├─► Stage 1: df.approxQuantile(features)
                │    Distributed clip threshold computation across all partitions
                │
                ├─► Stage 2: df.groupBy(bin, label).count()
                │    Each partition counts its own bin×label combinations
                │    → collect() to driver → MI computed from count table
                │
                └─► Stage 3: df.withColumn(feature, col + noise_expr)
                     Laplace noise via Spark SQL inverse-CDF expression
                     Runs as a JVM DAG — no Python loop over rows
```

**Local mode vs. cluster mode** — the code is identical:

| Target | What to change |
|---|---|
| Single machine (default) | Nothing — runs as `local[*]` using all CPU cores |
| YARN / Hadoop cluster | `.master("yarn")` in SparkSession |
| Standalone Spark cluster | `.master("spark://host:7077")` |

**Why this demonstrates Big Data relevance:**
- GS = (max − min) / n shrinks as n grows → less noise needed at scale
- Spark's Poisson mini-batch subsampling gives ε_amp = log(1 + q(e^ε − 1)) amplification
- The same DAG that processes 125,974 rows locally scales to 10M+ rows on a cluster with no code change

**Pandas fallback:** when `--no_spark` is passed or Java is not found, the same MI/variance/SNR
weighting and Laplace noise runs via pandas on a single core. Results are identical.

## All CLI Flags

