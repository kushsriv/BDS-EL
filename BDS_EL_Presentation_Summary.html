1. Subject Context — Big Data Systems

Big Data Systems is the study of how to collect, store, process, and analyze massive datasets — datasets so large that normal computers and software cannot handle them efficiently.

The challenge is not just storing this data — it is doing useful things with it while keeping it private and secure. This project sits at that intersection: How do you learn from big data without leaking sensitive information?

Key Big Data connections in this project:

  • Scale: NSL-KDD has 125,974 records with 43 features. Real networks generate millions of logs per day.
  • Privacy at scale: As dataset size grows, sensitivity (noise needed) shrinks — a fundamental Big Data property.
  • Composition across queries: Big Data analytics runs hundreds of queries over the same data — our PRV accountant makes this tractable under DP.
  • Local DP: Used in Apple iOS telemetry and Google RAPPOR — real Big Data deployments where centralized processing is impossible.

2. Problem Statement

The problem we are solving has three parts:

Part A — The Original Code Was Wrong

The starting codebase added fixed noise (sensitivity = 1.0) to every feature, ignoring what the actual data looked like. For some features this meant 250× too much noise (destroying accuracy). For others, not enough (not actually private). This is like using the same lock for a bicycle and a bank vault.

Part B — The "Budget Collapse" Problem Was Undocumented

Previous research only tested DP on 2–3 features. In a real IDS, you need all 37+ features. When the privacy budget ε=1.0 is split across 37 features, each feature gets ε/37 ≈ 0.027. This is too small for any signal — the model collapses. No prior work had systematically documented this collapse.

Part C — Privacy Accounting Was Wasteful

When you answer k queries, you must track total privacy cost. Standard methods overestimate this, making systems seem less privacy-friendly than they are. Tighter bounds = more queries = better utility.

Formal problem: How do you train a machine learning model on sensitive big data such that the model is (a) accurate enough to be useful AND (b) mathematically guaranteed not to leak private information?

3. What is Differential Privacy?

Differential Privacy (DP) is a mathematical guarantee invented at Microsoft Research (2006):

"Whether or not YOUR specific record is in the dataset, the output of any analysis looks approximately the same." — An attacker who sees the output cannot figure out if your data was used.

How It Works — Adding Calibrated Noise

Before sharing any result (e.g., "average failed login attempts"), add random mathematical noise. The noise is:

  • Large enough to hide any individual's contribution
  • Small enough that statistical patterns remain useful

6. Three Novel Contributions

1Clipped Sensitivity

Problem: Outlier values (a hacker running su 10,000 times) force enormous noise for everyone.

Solution: Clip at the 99th percentile before computing sensitivity. If 99% of values are below 5, treat 5 as the maximum.

Result: For su_attempted, noise reduces by 592,825,447×. Formally justified with a bounded bias proof.

Novelty: Prior work either uses raw sensitivity (too noisy) or clips without formal DP guarantees (not actually private). We provide both the technique and the proof.

2Importance-Weighted Budget Allocation

Problem: Splitting ε=1.0 equally across 37 features → ε/37 ≈ 0.027 per feature → model collapse.

Solution: Distribute budget proportional to each feature's Mutual Information (MI) with the attack label. Important features get more budget; unimportant ones get almost none.

Formula: εj = εtotal × (importancej / Σ importances)

Novelty: Provably ε-DP by basic composition theorem. Implemented for three weighting methods: MI-weighted, variance-weighted, SNR-optimal. Evaluated on real network data.

3PRV Accountant (Privacy Loss Random Variable)

Problem: After 100 queries at ε₀=0.5, naive accounting says total ε = 50. RDP says 31.5. Both are too loose.

Solution: Track the full distribution of privacy loss Z = log(Pr[output|record in] / Pr[output|record out]) and convolve k copies using FFT (Fast Fourier Transform) to get the exact composition.

Result: PRV gives ε = 15.34 at k=100 — 51.4% tighter than RDP. This means you can run twice as many queries within the same privacy budget.

Novelty: First implementation of Gopi et al. (2021) PRV theory for the Laplace mechanism on real network IDS data.

8. Research Paper Value

Why Reviewers Will Accept This

  1. Formal novelty with proofs. All three contributions have mathematical justification — they're not heuristics but provably correct techniques.
  2. Rigorous statistics. 30-run confidence intervals and Wilcoxon signed-rank tests — the kind of rigor IEEE journals require, rarely done in DP papers.
  3. Documented the budget collapse problem. First systematic study showing input perturbation fails on high-dimensional IDS data, with formal and experimental evidence.
  4. Real benchmark data. NSL-KDD is the standard IDS benchmark (125,974 real records), not synthetic experiments.
  5. MIA validation. 0/24 bound violations confirms DP guarantees hold in practice, not just in theory.
Paper narrative: "Standard input perturbation completely fails on realistic feature counts (37 features → 53.3% accuracy). We identify the root cause (per-feature budget collapse), propose three formally-proven solutions, and demonstrate near-baseline accuracy (94.70% at ε=1.0) with 30-run statistical validation."

9. Future Work & Suggestions

Short Term (1–2 weeks)

  1. UNSW-NB15 cross-validation — Dataset adapter already built in multi_dataset.py. Just download the CSV from the UNSW research website and run python main.py --dataset UNSW_NB15_training-set.csv --no_spark. Cross-dataset validation is one of the strongest arguments for generalizability.
  2. Multi-class DP-SGD — Module 11 shows 5-class IDS collapses with input perturbation. Extending DP-SGD to softmax (5-class output) directly fixes this and completes the story.
  3. Hyperparameter sensitivity — Show results across clipping norms C and noise multipliers σ. Reviewers will always ask this.

Medium Term (1 month)

  1. Federated Learning extension — Multiple organizations (e.g., university networks) each train locally using DP-SGD and share only gradients. This is the real-world deployment model and is currently one of the hottest research directions in privacy.
  2. CIC-IDS-2017 dataset — More modern attacks (2017 web attacks). Adapter already built.
  3. Adaptive ε selection — Automatically find the smallest ε that maintains >95% accuracy. Practically useful for system administrators.

Long Term (PhD-level)

  1. Ablation study — Measure impact of each novel contribution independently to quantify individual value.
  2. Compare with Opacus / TF Privacy — Benchmark against Meta's and Google's DP libraries to show competitiveness with industry standards.
  3. Differentially Private Transformer for IDS — Apply DP to attention mechanisms in neural networks for network traffic analysis — frontier research.
